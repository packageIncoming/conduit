1   Project Overview

Conduit is an HTTP/1.1 web server written from scratch in C, targeting Linux. It progresses through eight milestones (TRD-00 through TRD-07), each building a testable vertical slice of functionality. The final artifact is a concurrent, epoll-driven, thread-pooled static file server with honest benchmark data and a professional README suitable for a systems engineering portfolio.

1.1   Design Philosophy

  • Learn by building. Every syscall, every struct, every edge case is written by hand. AI may be used for post-completion code review and conceptual questions only — never for writing or debugging implementation code.
  • Test-driven milestones. Each TRD ships with an automated bash test suite. Green suite = milestone complete. No exceptions.
  • Honest engineering. Benchmark results are framed as local loopback measurements, not production performance claims.

2   Individual TRD Plans

TRD-00  —  TCP Echo: Foundation
Build a single-threaded TCP server that listens on a configurable port, accepts one connection at a time, reads incoming bytes (without parsing), responds with a hardcoded HTTP/1.1 200 OK, and loops.

Core Concepts

  • POSIX socket API: socket(), bind(), listen(), accept(), read(), write(), close()
  • SO_REUSEADDR — why it matters for server restarts during development
  • File descriptors: listening fd vs. client fd lifecycle
  • sockaddr_in struct, htons(), INADDR_ANY
  • HTTP at the byte level: status line + headers + blank line + body
  • Content-Length — bytes, not characters (UTF-8 multibyte awareness)

Requirements Scope

  • Port from command-line argument; usage message + exit(1) if missing
  • SO_REUSEADDR set on listen socket
  • Backlog ≥ 128
  • Accept loop: accept → read (discard) → write hardcoded response → close client fd → loop
  • Descriptive perror() on every syscall failure
  • Failed accept() logs and continues (no crash)

Comprehension Focus

Why SO_REUSEADDR? What is the listen backlog and what happens when it fills? Why does the server need two file descriptors? What is INADDR_ANY vs. binding to a specific IP?

Key Resources

Beej's Guide to Network Programming (Ch. 1–6), man 2 socket/bind/listen/accept, Cloudflare TCP backlog blog post.

TRD-01  —  HTTP/1.1 Request Parsing
Parse the HTTP request line and headers from raw bytes into a structured representation. Route based on method and path. Return appropriate status codes for valid and malformed requests.

Core Concepts

  • HTTP/1.1 request anatomy: request line (METHOD SP URI SP VERSION CRLF), headers (Key: Value CRLF), blank line terminator (CRLF CRLF)
  • C string parsing: strstr(), strtok_r(), sscanf(), manual pointer arithmetic
  • Buffer management: reading into a fixed buffer, detecting incomplete reads, handling requests that arrive across multiple read() calls
  • Request struct design: method enum, path string, header key-value array
  • Status codes: 200 (OK), 400 (Bad Request), 405 (Method Not Allowed), 404 (Not Found)
  • RFC 9110 / RFC 9112 — the authoritative source for HTTP semantics and message format

Requirements Scope

  • Define http_request_t struct: method, path, version, headers array (key-value pairs), header count
  • Parse request line: extract method, URI, HTTP version
  • Parse headers: populate key-value array until \r\n\r\n
  • Return 400 for malformed request lines (missing fields, bad version string, no CRLF)
  • Return 405 for unsupported methods (only GET required)
  • Return 404 for unrecognized paths (routing to be expanded in TRD-02)
  • Return 200 with body for a known test path (e.g., / returns a hardcoded string)

Hardest Part

Partial reads. A single read() call is not guaranteed to return the complete request. The parser must detect when it has received the full header block (found \r\n\r\n) and handle the case where it hasn't. This is the first real encounter with TCP's stream nature.

Comprehension Focus

Why can't you assume one read() = one complete request? What happens if the client sends a header with no colon? Why use strtok_r() instead of strtok()? What is the difference between \n and \r\n in HTTP?

Key Resources

RFC 9112 §2–3 (message format), RFC 9110 §9 (methods), Beej's Guide §7 (advanced techniques for partial reads).

TRD-02  —  Static File Serving
Resolve HTTP request paths to filesystem paths and serve files with correct Content-Type and Content-Length headers. Handle missing files and prevent directory traversal attacks.

Core Concepts

  • POSIX file I/O: open(), read(), close(), fstat() for file size
  • MIME type mapping: extension → Content-Type (at minimum: .html, .css, .js, .txt, .jpg, .png)
  • Path resolution: mapping URI to a document root directory
  • Directory traversal prevention: why ../ in a URI is dangerous, how realpath() can validate resolved paths stay within the document root
  • Content-Length from fstat() — getting the exact byte count before sending

Requirements Scope

  • Accept a document root directory as a second command-line argument: ./conduit <port> <docroot>
  • Map request path to docroot + path; serve index.html if path is /
  • Use fstat() for Content-Length; set Content-Type based on file extension
  • Return 404 if file does not exist
  • Return 403 if resolved path escapes document root (traversal attempt)
  • Read file into buffer and write as response body

Hardest Part

Directory traversal is the security-critical requirement. A naive implementation of docroot + path is vulnerable to GET /../../../etc/passwd. The canonical solution is: resolve the full path with realpath(), then verify the result starts with the document root prefix. Getting this wrong is a real CVE-class bug.

Comprehension Focus

Why is realpath() necessary instead of just checking for .. in the raw URI? What happens if you send a Content-Length that doesn't match the actual body size? Why does the server need a separate MIME mapping instead of always sending text/plain?

Key Resources

man 2 open, man 2 fstat, man 3 realpath, OWASP Path Traversal overview, Mozilla MDN MIME types reference.

TRD-03  —  HTTP Response Engine
Replace ad-hoc response construction with a reusable, structured response layer. All HTTP responses flow through a single codepath.

Core Concepts

  • Response struct design: status code, reason phrase, headers array, body pointer + body length
  • Dynamic memory in C: malloc()/free() for variable-length header assembly
  • Wire-format serialization: status line → headers → \r\n → body, written to socket fd
  • Status code semantics: 200, 400, 403, 404, 405, 500
  • Separation of concerns: response building vs. I/O

Requirements Scope

  • Define http_response_t struct: status code, reason string, headers (key-value array), body pointer, body length
  • Implement: response_set_status(), response_add_header(), response_set_body()
  • Implement response_send(fd, response) — serializes struct to wire format and writes to socket
  • Refactor all existing response paths (TRD-01 and TRD-02) to use the new engine
  • Return proper 400/403/404/405/500 responses through the engine
  • Automatically set Content-Length from body length if body is present

Hardest Part

Memory ownership. Who allocates the body buffer? Who frees it? If the response engine mallocs a serialization buffer, when does it get freed if the send is deferred (which it will be in TRD-04 with epoll)? Designing the ownership model now prevents use-after-free bugs later.

Comprehension Focus

Why build a response engine instead of writing bytes directly? What happens if Content-Length is wrong? Why must headers be written before the body with only \r\n\r\n separating them? What does a 500 response mean and when should the server send one?

Key Resources

RFC 9110 §15 (status codes), RFC 9112 §6 (response format), man 3 malloc/man 3 free.

TRD-04  —  Epoll Event Loop   ⚠ HARDEST TRD
Replace the blocking single-connection model with a non-blocking epoll-driven event loop capable of handling many concurrent connections.

Core Concepts

  • Blocking vs. non-blocking I/O: what O_NONBLOCK does at the kernel level
  • epoll_create1(), epoll_ctl() (ADD/MOD/DEL), epoll_wait()
  • Event flags: EPOLLIN, EPOLLOUT, EPOLLERR, EPOLLHUP
  • Edge-triggered (EPOLLET) vs. level-triggered — why ET requires draining the fd until EAGAIN
  • EAGAIN / EWOULDBLOCK — what they signal and how to handle them
  • Per-connection state machine: READING_REQUEST → SENDING_RESPONSE → DONE
  • Partial sends: send() may not write all bytes; must track offset

Requirements Scope

  • Set listen socket to O_NONBLOCK
  • Create epoll instance; register listen socket for EPOLLIN
  • epoll_wait() loop: dispatch events by fd type (listener vs. connection)
  • On new connection: accept() in a loop (edge-triggered may deliver multiple), set O_NONBLOCK, register with epoll
  • On readable: feed bytes to request parser; on complete request, process and stage response
  • On writable: flush pending send buffer; handle partial sends via offset tracking
  • Per-connection struct: read buffer, write buffer + offset, state enum, fd
Figure 2 — Per-connection state machine in the epoll event loop

Hardest Part

Edge-triggered mode. In ET mode, epoll only notifies once per state change. If you don't drain all available data from a readable fd (looping read() until EAGAIN), you will never get another notification and the connection silently hangs. This is the single most common epoll bug and the hardest to debug because it manifests as "sometimes works, sometimes doesn't."

Comprehension Focus

Why does the event loop must never block on any individual connection? What is the difference between edge-triggered and level-triggered? Why must you accept in a loop under ET? Why can't you just send() the full response in one call?

Key Resources

man 7 epoll (read the entire page — it's short and critical), man 2 epoll_create1, man 2 epoll_ctl, man 2 epoll_wait, man 2 fcntl (for O_NONBLOCK). The epoll man page's "Suggested usage" section is essentially the algorithm.

TRD-05  —  Thread Pool
Offload request processing from the event loop thread to a fixed pool of worker threads using a producer-consumer task queue.

Core Concepts

  • pthread_create(), pthread_join()
  • Mutex (pthread_mutex_t) — protecting shared data structures
  • Condition variable (pthread_cond_t) — signaling between event loop and workers
  • Producer-consumer pattern: event loop enqueues tasks, worker threads dequeue
  • Task struct design: what data must be passed from the event loop to a worker
  • Thread-safe queue: linked list or ring buffer protected by mutex
  • Graceful shutdown: drain queue, set stop flag, broadcast condvar, join all threads

Requirements Scope

  • Fixed thread count (N configurable via #define or command-line argument)
  • Task queue: linked list with mutex protection
  • threadpool_init(n), threadpool_enqueue(task), threadpool_destroy()
  • Worker loop: lock → wait on condvar → dequeue → unlock → execute → repeat
  • Shutdown flag: set flag + pthread_cond_broadcast() to wake all workers; workers exit when flag is set and queue is empty
  • Event loop calls threadpool_enqueue() on completed request parse
  • Workers call response_send() directly (fd passed in task struct)
Figure 3 — Producer-consumer thread pool architecture

Hardest Part

Correct condvar usage. Condition variables can produce spurious wakeups — the pthread_cond_wait() call must be inside a while loop that re-checks the predicate, not an if. The shutdown race is the other trap: if you set the stop flag but don't broadcast the condvar, sleeping workers never wake up. If you broadcast but workers don't check the flag, they block forever on an empty queue.

Comprehension Focus

Why a while loop around pthread_cond_wait() instead of if? What happens if two workers dequeue the same task? Why must the mutex be held when calling pthread_cond_wait()? How does fd ownership transfer from the event loop to a worker thread?

Key Resources

man 3 pthread_create, man 3 pthread_mutex_lock, man 3 pthread_cond_wait, OSTEP Ch. 30 (Condition Variables).

TRD-06  —  Hardening
Make the server robust against malformed input, resource exhaustion, and ungraceful termination. After this TRD, the server should not crash under hostile conditions.

Core Concepts

  • Max request size enforcement — reject oversized requests with 413 (Payload Too Large)
  • Connection limits — capping concurrent open fds; returning 503 when at capacity
  • Connection timeouts — closing idle connections without blocking the event loop (timestamp per connection, sweep on each epoll_wait() return)
  • SIGPIPE — writing to a closed socket raises this signal; suppress with signal(SIGPIPE, SIG_IGN) or MSG_NOSIGNAL flag on send()
  • Graceful shutdown: SIGINT/SIGTERM → set flag → stop accepting → drain thread pool → close all fds
  • Async-signal-safe functions — what you can and cannot do inside a signal handler
  • Error propagation: every recv()/send()/open() return value checked

Requirements Scope

  • Enforce max request buffer size (e.g., 8 KB headers); return 400 if exceeded
  • Track open connection count; refuse new connections at limit with 503
  • Register SIGPIPE as SIG_IGN at startup
  • Install SIGINT/SIGTERM handler that sets a volatile sig_atomic_t shutdown flag
  • Shutdown sequence: stop accepting → drain thread pool → close all connections → close listen socket → exit(0)
  • Idle connection timeout (configurable, e.g., 30 seconds)
  • Return 400 for any request that fails parsing (already partial from TRD-01; formalize here)

Hardest Part

Timeout implementation without blocking. You cannot call sleep() in the event loop. The pattern is: store a last_active timestamp on each connection struct, pass a timeout to epoll_wait(), and on each return sweep all connections for expired timestamps. This is how production servers implement idle timeouts.

Comprehension Focus

Why is SIGPIPE fatal by default? What functions are async-signal-safe and why does that matter? Why is volatile sig_atomic_t necessary for the shutdown flag instead of a plain int? How does the timeout sweep interact with epoll?

Key Resources

man 7 signal, man 7 signal-safety, man 2 epoll_wait (timeout parameter), OSTEP Ch. 26 (Concurrency introduction).

TRD-07  —  Benchmarking & Documentation
Measure the server's performance honestly, audit for memory leaks, and produce a professional README that positions Conduit as a portfolio artifact for systems engineering roles.

Core Concepts

  • Benchmarking methodology: warm-up runs, controlling variables, multiple trials, statistical reporting
  • wrk — HTTP benchmarking tool: connections, threads, duration, latency percentiles
  • Throughput (req/sec) vs. latency (p50/p95/p99) — why p99 matters more than average for servers
  • Comparison baseline: Python http.server serving the same static file
  • valgrind --tool=memcheck — detecting memory leaks and use-after-free
  • README structure for a systems project: architecture overview, build instructions, benchmark results, design decisions, known limitations

Requirements Scope

  • Run wrk at varying concurrency levels (1, 10, 50, 100 connections) against Conduit
  • Record req/sec and p99 latency per concurrency level in a table
  • Run same benchmark against Python http.server; produce comparison table
  • Run valgrind --tool=memcheck under load; fix any leaks found
  • Write README: project summary, architecture diagram (ASCII or SVG), build/run instructions, benchmark table with methodology notes, design decisions, known limitations
  • Tag final commit; confirm GitHub repo is clean and public
Honesty requirement: Frame benchmark results as "throughput under N concurrent connections on local loopback on [hardware specs]." Do not claim Conduit outperforms nginx in production. The comparison against Python http.server is fair and demonstrates real gains; a comparison against nginx is useful for learning but should be contextualized honestly. Recruiters who know systems will respect accurate framing more than inflated claims.

Hardest Part

Interpreting results. Benchmarks on a single dev machine with loopback networking eliminate network latency entirely, which inflates req/sec numbers. Understanding what the numbers actually mean — and being able to explain that in an interview — is the real deliverable of this TRD.

Comprehension Focus

Why does p99 latency matter more than average? What does wrk's "transfer/sec" actually measure? Why would Conduit lose to nginx at high concurrency? What do valgrind's "definitely lost" vs. "still reachable" categories mean?

Key Resources

wrk GitHub repo + README, man valgrind, Brendan Gregg's Systems Performance (benchmarking methodology sections).

4   TRD Package Format

Every TRD ships as exactly four artifacts:

# Document Description Format
1 Prior Knowledge Higher-level concepts, mental models, external resource links. Read before or during implementation. Printable HTML
2 TRD Objective, technical requirements, acceptance criteria table, comprehension questions, next-TRD preview. Inline links to Hints doc. Printable HTML
3 Hints Two-tier progressive guidance per requirement. Level 1 = nudge. Level 2 = stronger push. 15-minute rule before escalating. Printable HTML
4 Test Suite Automated bash script. Compiles, starts server, runs tests, reports pass/fail with color output. Bash script

All printable documents: professional layout, black ink on white, headings, tables, diagrams where necessary. De-personalized — tied to the Conduit project, not to any individual.

Critical bash note: Under set -euo pipefail, bare ((VAR++)) exits non-zero when the variable is 0. Always use VAR=$((VAR + 1)) syntax in test suites.
