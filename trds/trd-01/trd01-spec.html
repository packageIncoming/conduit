TRD-01: HTTP/1.1 Request Parsing

Conduit HTTP Server Project
Depends on: TRD-00  |  Difficulty: Medium  |  Estimated: 4–7 sessions

Objective

Parse the HTTP request line and headers from raw TCP bytes into a structured http_request_t. Route by method and path. Return correct status codes for valid, malformed, and unsupported requests.

Technical Requirements

R1 — Request Struct

Define an http_request_t struct containing: method, path, HTTP version, an array of header key-value pairs, and a header count.

Stuck? → Hints H1

R2 — Request Line Parsing

Extract method, URI, and HTTP version from the first line of the request. The format is METHOD SP URI SP VERSION CRLF. Only HTTP/1.1 is accepted as a valid version.

Stuck? → Hints H2

R3 — Header Parsing

Parse each subsequent line as a Key: Value header pair until a bare \r\n is encountered (marking the end of the header block). Store parsed headers in the request struct.

Stuck? → Hints H3

R4 — End-of-Headers Detection

Accumulate bytes from read() into a buffer. After each read, check for the \r\n\r\n sequence. Do not attempt to parse until the full header block has arrived. Handle the partial-read case (see Prior Knowledge §2).

Stuck? → Hints H4

R5 — Status Code Routing

  • 200 OK — Valid GET request to / (return a hardcoded plain-text body)
  • 400 Bad Request — Malformed request line (missing fields, bad version, garbage input)
  • 404 Not Found — Valid GET to any path other than /
  • 405 Method Not Allowed — Any non-GET method
Stuck? → Hints H5

R6 — Response Format

All responses must include: a valid HTTP/1.1 status line, a Content-Type header, a Content-Length header whose value matches the actual body byte count, and the body (which may be empty for error responses, or a short error message). This is still ad-hoc construction — the response engine comes in TRD-03.

Stuck? → Hints H6

R7 — Server Loop

The server must continue accepting and handling new connections after each request (same accept loop as TRD-00). One connection, one request, one response, close, loop.

Post-Completion Comprehension

Answer these before requesting TRD-02. Write them out — don't just think "I know this."

  1. Why can't a single read() be assumed to contain a complete HTTP request?
  2. What does \r\n\r\n signify and why is it the correct sentinel for "headers are done"?
  3. Why use strtok_r() instead of strtok()? What will break later if you use the non-reentrant version?
  4. A header like Host: localhost:9876 has a colon in the value. How does the parser handle this?
  5. What happens to the connection if the server receives a partial request and never gets the rest? (Think about what TRD-06 will fix.)
  6. Why does the test suite send raw bytes via nc for malformed-request tests instead of using curl?

Preview: TRD-02

The server currently returns hardcoded text. TRD-02 replaces this with real file serving: mapping URI paths to filesystem paths, reading files with open()/fstat()/read(), setting Content-Type from file extensions, and blocking directory traversal attacks.
