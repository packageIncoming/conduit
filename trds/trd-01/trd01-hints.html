TRD-01 Hints

Progressive Guidance — Conduit HTTP Server Project
Level 1 = nudge  |  Level 2 = stronger push  |  Try Level 1 for 15+ minutes before escalating

H1 — Request Struct Design (R1)

Level 1

Think about what fields the rest of the code needs to access after parsing. The method determines the response logic. The path determines what to serve. The version determines validity. Headers are key-value pairs with an unknown (but bounded) count. Start with fixed-size character arrays — dynamic allocation is not needed yet.

Level 2

Use char method[8], char path[1024], char version[16]. For headers, use a struct array: struct { char key[256]; char value[512]; } headers[32] and an int header_count. Initialize header_count to 0 before parsing. These sizes are generous — a real production server would be more careful, but correctness is the goal now.

H2 — Request Line Parsing (R2)

Level 1

The request line ends at the first \r\n. Isolate that line first, then split it into three parts. There should be exactly three space-separated tokens. If there aren't three, it's malformed.

Level 2

One approach: use strstr(buf, "\r\n") to find the end of the first line. Null-terminate there temporarily. Then use sscanf(line, "%7s %1023s %15s", req.method, req.path, req.version) — the return value tells you how many fields matched. If it's not 3, return 400. Then validate strcmp(req.version, "HTTP/1.1") == 0. If not, return 400.

H3 — Header Parsing (R3)

Level 1

After the request line, each header occupies one line (terminated by \r\n). The key and value are separated by the first colon — not just any colon. Think about Host: localhost:9876.

Level 2

Walk a pointer from the end of the request line. For each line: find the next \r\n, that's the end of this header. Within that line, find the first : using strchr(). Everything before the colon is the key; everything after (skip optional leading whitespace) is the value. Copy both into headers[header_count] and increment. Stop when you hit a line that's just \r\n.

H4 — End-of-Headers Detection / Partial Reads (R4)

Level 1

Your read() call should be inside a loop. Each iteration appends to the buffer. After each append, call strstr(buffer, "\r\n\r\n"). If it returns non-NULL, the full header block has arrived and you can parse. If NULL, keep reading.

Level 2

Pseudocode for the read loop:

char buf[4096];
int total = 0;
while (total < sizeof(buf) - 1) {
    int n = read(client_fd, buf + total, sizeof(buf) - 1 - total);
    if (n <= 0) break;
    total += n;
    buf[total] = '\0';
    if (strstr(buf, "\r\n\r\n")) break;
}

The -1 reserves space for the null terminator. If the loop exits without finding \r\n\r\n, the request is either incomplete (client disconnected) or oversized — both can be treated as bad requests for now.

H5 — Status Code Routing (R5)

Level 1

Think about the order of checks. Some conditions should be checked before others. A malformed request line should be caught before you check the method or path. A wrong method should be caught before you check the path.

Level 2

Evaluation order: (1) Did the request line parse into three valid fields? No → 400. (2) Is the version HTTP/1.1? No → 400. (3) Is the method GET? No → 405. (4) Is the path /? No → 404. (5) Everything valid → 200.

H6 — Response Construction (R6)

Level 1

Build the response as a single string using snprintf(). Status line first, then headers, then \r\n, then body. Compute the body length before building the response so you can set Content-Length accurately. Remember: bytes, not characters.

Level 2
const char *body = "Conduit is alive";
int body_len = strlen(body);
char resp[4096];
int resp_len = snprintf(resp, sizeof(resp),
    "HTTP/1.1 200 OK\r\n"
    "Content-Type: text/plain\r\n"
    "Content-Length: %d\r\n"
    "\r\n"
    "%s",
    body_len, body);
write(client_fd, resp, resp_len);

For error responses, follow the same pattern but change the status line and body. A 404 response still needs valid headers and a body (even if the body is just "Not Found").
