1   Anatomy of an HTTP/1.1 Request

An HTTP request is a structured text message sent over the TCP byte stream established in TRD-00. It has three parts:

Figure 1 — Structure of an HTTP/1.1 request as raw bytes on the wire

1.1   The Request Line

The first line is always: METHOD SP URI SP VERSION CRLF where SP is a single space character (0x20) and CRLF is \r\n (bytes 0x0D 0x0A). For TRD-01, the only method that must be accepted is GET. The only version that must be accepted is HTTP/1.1.

Parsing this line means extracting three strings: the method, the URI path, and the version. If any piece is missing or the version string is wrong, the request is malformed.

1.2   Headers

After the request line, zero or more headers follow. Each header is: Key: Value\r\n. The key and value are separated by a colon, optionally followed by whitespace. Headers end when a bare \r\n appears on its own — making the full terminator sequence \r\n\r\n (the last header's CRLF plus the blank line's CRLF).

For TRD-01, headers must be parsed and stored, but their values don't need to influence behavior yet (no routing based on Host, no Content-Length body reading). They just need to be correctly extracted into a data structure.

1.3   The \r\n\r\n Sentinel

Detecting \r\n\r\n in the receive buffer is how the server knows the complete header block has arrived. This is the most important parsing operation in TRD-01. Until this sequence is found, the server has not received a complete request and must continue reading.

2   TCP Is a Stream, Not Messages

This was mentioned in TRD-00's Prior Knowledge, but it becomes critical now. A single read() call on a TCP socket is not guaranteed to return one complete HTTP request. Three scenarios:

Scenario What happens What the server must do
Complete request in one read Buffer contains everything up to and including \r\n\r\n Parse immediately
Partial request Buffer has the start of the request but no \r\n\r\n yet Keep buffer contents, read() again, append to buffer
Multiple requests in one read Buffer contains more than one full request (pipelining) Not required for TRD-01 — handle one request per connection

The partial-read case is the one that matters. The implementation pattern is: maintain a buffer and an offset tracking how many bytes have been received, call read() in a loop, and after each read check whether \r\n\r\n has appeared in the accumulated buffer.

3   C String Parsing Toolkit

There are several approaches to parsing the request line and headers in C. None is "correct" — pick the one that makes sense.

Function Use for Watch out for
strstr() Finding \r\n\r\n in the buffer Returns NULL if not found — always check
strtok_r() Splitting the request line by spaces Use _r (reentrant), not strtok() — the non-reentrant version uses global state and will break in TRD-05 with threads
sscanf() Extracting method, path, version from request line Check the return value (number of fields matched)
strchr() Finding the colon separator in header lines Headers can have colons in their values (e.g., Host: localhost:9876)
Manual pointer arithmetic Walking through the buffer byte by byte Off-by-one errors, null terminator placement
Key decision: The receive buffer from read() is not null-terminated. Before using any str*() function, you must either null-terminate the buffer at the number of bytes received, or use memchr()/memmem() which operate on counted bytes rather than C strings.

4   Struct Design

The parsed request needs a home. A typical approach:

typedef struct {
    char method[8];          /* "GET", "POST", etc. */
    char path[1024];         /* "/index.html" */
    char version[16];        /* "HTTP/1.1" */
    struct {
        char key[256];
        char value[512];
    } headers[32];
    int header_count;
} http_request_t;

Fixed-size buffers are fine for TRD-01. The sizes above are reasonable defaults. Dynamic allocation adds complexity with no learning payoff at this stage — save that for TRD-03.
