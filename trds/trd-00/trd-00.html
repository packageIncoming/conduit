TRD-00: TCP Echo

Conduit Foundation — Conduit HTTP Server Project

Milestone Summary

Objective: Build a single-threaded TCP server in C that listens on a configurable port, accepts one connection at a time, reads an incoming HTTP request (without parsing it), and responds with a hardcoded HTTP/1.1 200 OK response containing a plain text body. Then closes the connection and loops to accept the next one.

What this is NOT: This is not an HTTP server yet. There is no request parsing, no routing, no file I/O, no concurrency. This is a raw TCP accept loop that happens to speak enough HTTP to be testable with curl.

Why this matters: Every line of Conduit's future architecture — epoll, thread pool, request parsing — sits on top of this socket foundation. If the bind/listen/accept/read/write/close cycle has bugs, leaks FDs, or mishandles errors, debugging ghosts will persist for weeks in later milestones. Get this bulletproof now.

Technical Requirements

R1: Socket Initialization

  • Create a TCP socket using socket(AF_INET, SOCK_STREAM, 0).
  • Set SO_REUSEADDR via setsockopt so the server can restart without TIME_WAIT blocking the port.
  • Bind to INADDR_ANY on a port specified as a command-line argument (./conduit <port>).
  • If no port argument is provided, print a usage message to stderr and exit with code 1.
  • If socket(), setsockopt(), bind(), or listen() fails, print a descriptive error to stderr (use perror() or equivalent) and exit with a non-zero code.
Stuck? See Hints document — H1 (socket setup) and H2 (SO_REUSEADDR).

R2: Listen and Accept Loop

  • Call listen() with a backlog of at least 128.
  • Enter a loop that calls accept() to get a connected client file descriptor.
  • If accept() fails, log the error to stderr and continue (do NOT crash the server on a failed accept).
Stuck? See Hints document — H3 (the accept loop structure).

R3: Request Reading

  • Read from the client socket using read() or recv() into a stack-allocated buffer of at least 4096 bytes.
  • Multi-read assembly, partial reads, and chunked encoding do NOT need handling at this stage. A single read() call is sufficient.
  • If read() returns 0 (client closed connection) or -1 (error), close the client FD and continue the accept loop.
Stuck? See Hints document — H4 (read behavior and edge cases).

R4: Hardcoded HTTP Response

Regardless of what was read, respond with exactly this:

HTTP/1.1 200 OK\r\n
Content-Type: text/plain\r\n
Content-Length: [body length in bytes]\r\n
Connection: close\r\n
\r\n
Conduit is alive — TRD00
  • The Content-Length MUST match the actual body length in bytes, not characters. The em dash (—) is a multi-byte UTF-8 character. Count bytes, not glyphs. Use echo -n "your string" | wc -c to verify.
  • Use write() or send() to transmit the full response.
  • After sending the response, close the client file descriptor with close().
Stuck? See Hints document — H5 (building the response string) and H6 (byte counting).

R5: Clean Shutdown

  • Graceful SIGINT/SIGTERM handling is NOT required at this milestone. Ctrl+C to kill is acceptable.
  • The listening socket must not leak FDs in the normal accept loop path.

R6: Compilation

The project MUST compile with:

gcc -Wall -Wextra -Werror -std=c11 -o conduit conduit.c

Zero warnings. -Werror means every warning is a build failure. Non-negotiable for every TRD going forward.

Stuck? See Hints document — H7 (common warnings and how to fix them).

R7: No External Dependencies

Only POSIX headers and the C standard library. No third-party libraries.

Concepts to Understand After Completion

Before moving to TRD-01, the following should be explainable without looking anything up:

  1. What does socket(AF_INET, SOCK_STREAM, 0) actually return, and what does the kernel do when it's called?
  2. Why is SO_REUSEADDR needed, and what happens without it when restarting the server quickly?
  3. What is the listen() backlog, and what happens when it fills up?
  4. What's the difference between the listening FD and the client FD returned by accept()?
  5. Why does Content-Length matter, and what breaks if it's wrong?
  6. What does Connection: close tell the client?
  7. What happens if close() is never called on the client FD in the loop?
If any of these can't be answered confidently, TRD-00 isn't done regardless of whether the tests pass.
