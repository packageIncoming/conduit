TRD-00 Prior Knowledge

TCP & POSIX Sockets — Conduit HTTP Server Project

This document covers foundational concepts needed before or during TRD-00 implementation. It's not a tutorial — it's a map. Read enough to orient, then learn the rest by writing code and hitting walls.

1. The TCP/IP Mental Model

Conduit is a server. It sits and waits for other programs (clients) to initiate a conversation. TCP provides the rules: a reliable, ordered, byte-stream connection between two endpoints.

The key thing to internalize: TCP has no concept of "messages." It's a stream of bytes. HTTP imposes message structure on top of that stream (headers, body, Content-Length). At TRD-00, there's no need to care about HTTP structure yet — just read some bytes and write some bytes. But awareness of this distinction matters, because it will bite hard in TRD-01 when determining where one HTTP message ends and another begins.

The connection lifecycle from the server's perspective:

  1. Create a socket (a file descriptor the OS uses to track network state)
  2. Bind that socket to an address + port ("I own port 9876")
  3. Listen, telling the OS to queue incoming connections
  4. Accept a connection from the queue, getting a new file descriptor for that specific client
  5. Read from / write to that client FD
  6. Close the client FD

This is the socket → bind → listen → accept → read/write → close pipeline. Every networked C program follows this pattern or some variation of it.

2. File Descriptors — The UNIX Abstraction

In UNIX/Linux, everything is a file. A socket is a file. A pipe is a file. /dev/null is a file. The OS returns an integer — a file descriptor (FD) — used to interact with these things through a uniform interface (read(), write(), close()).

Calling socket() returns an FD for the listening socket. Calling accept() returns a different FD for each connected client. Forgetting to close() a client FD leaks it. The OS has a per-process FD limit (usually 1024 by default). Leak enough FDs and the server silently stops accepting new connections.

Key concept — the two FDs:

listening_fd = socket(...)   -- Lives for the lifetime of the server
client_fd    = accept(...)   -- New one per connection. Close it when done.

This distinction trips up beginners. Never read from or write to the listening FD. It's a doorway. The client FD is the actual conversation.

3. The struct sockaddr_in Dance

To bind a socket to an address, fill out a struct sockaddr_in and pass it to bind(). This struct tells the OS: "Listen on this IP, on this port."

The annoying parts:

  • htons() (host-to-network short): Network protocols use big-endian byte order. Most machines use little-endian. htons() converts a port number to network byte order. Forget it and the socket binds to a garbage port.
  • INADDR_ANY: Means "listen on all network interfaces." Standard for dev servers.
  • Casting: bind() takes a struct sockaddr *, but the code fills out a struct sockaddr_in. Cast the pointer. This is one of C's ugly corners — accept it and move on.

The typical setup:

struct sockaddr_in addr;
memset(&addr, 0, sizeof(addr));        // zero the struct
addr.sin_family = AF_INET;             // IPv4
addr.sin_addr.s_addr = INADDR_ANY;     // all interfaces
addr.sin_port = htons(port);           // port, byte-swapped

Don't memorize this. Understand what each line does, then copy it from prior code every time.

4. SO_REUSEADDR — Why It's Needed

When a TCP connection closes, the OS puts the port into a TIME_WAIT state for typically 60 seconds. This prevents stale packets from a previous connection being misinterpreted as belonging to a new one.

The side effect: killing and immediately restarting the server causes bind() to fail with EADDRINUSE ("Address already in use") because the port is still in TIME_WAIT.

setsockopt(fd, SOL_SOCKET, SO_REUSEADDR, ...) tells the OS: "The port might be in TIME_WAIT. Allow binding anyway." Safe for development and standard practice for servers.

5. listen() Backlog

listen(fd, backlog) marks the socket as passive (willing to accept incoming connections) and sets the size of the connection queue.

When a client does a TCP handshake, the kernel completes it and puts the connection into a queue. accept() pulls connections out. The backlog parameter hints at maximum queue length.

If the queue fills up (because accept() is called too slowly), new clients get refused or retry. For TRD-00 with a single-threaded sequential server, a backlog of 128 is more than enough. This becomes important in TRD-04/05 with epoll and a thread pool.

6. HTTP/1.1 at the Byte Level

TRD-00 doesn't parse HTTP. But it writes an HTTP response, so the byte-level structure matters.

An HTTP response on the wire:

HTTP/1.1 200 OK\r\n
Header-Name: header-value\r\n
Another-Header: value\r\n
\r\n
[body bytes]

Key points:

  • Every line ends with \r\n (CRLF — bytes 0x0D 0x0A). Not just \n. Specified in RFC 7230.
  • Headers end with a blank line: \r\n\r\n. This double-CRLF signals the client that body is starting.
  • Content-Length tells the client exactly how many bytes the body is. Wrong value = client hangs or truncates.
  • Connection: close tells the client not to reuse the connection. Simplifies TRD-00.

Byte counting the TRD-00 body

The body: Conduit is alive — TRD00

  • "Conduit is alive " = 17 bytes (ASCII, 1 byte each, including trailing space)
  • "—" = 3 bytes (UTF-8 encoding of U+2014 EM DASH: 0xE2 0x80 0x94)
  • " TRD00" = 6 bytes (space + 5 ASCII chars)
  • Total: 17 + 3 + 6 = 26 bytes
Never trust documentation blindly. Count bytes independently. Use echo -n "your string" | wc -c in the terminal to verify. In production, a wrong Content-Length causes real outages.

7. Error Handling in C — The perror() Pattern

Every syscall in TRD-00 can fail. In C, these functions return -1 on failure and set the global errno variable.

The standard pattern:

if (bind(fd, (struct sockaddr *)&addr, sizeof(addr)) == -1) {
    perror("bind");   // prints "bind: [human-readable error]"
    close(fd);
    exit(EXIT_FAILURE);
}

perror() reads errno and prints the corresponding message. Use it for every syscall. Check return values. Print with perror(). Exit or continue as appropriate.

8. C Compilation Flags

Required: gcc -Wall -Wextra -Werror -std=c11 -o conduit conduit.c

  • -Wall: Most common warnings (unused variables, implicit declarations, etc.)
  • -Wextra: Additional warnings beyond -Wall (unused parameters, sign comparison, etc.)
  • -Werror: Every warning becomes a compilation error. Build fails on a single warning.
  • -std=c11: Use the C11 standard.

This is strict on purpose. Serious systems shops compile with at least -Wall -Werror. Writing warning-free C from the start prevents sloppy habits.

9. Debugging Tools

  • curl — Primary HTTP client for testing. curl -v http://localhost:9876/ shows request and response headers. The -v flag is critical.
  • nc (netcat) — Raw TCP client. Useful for edge cases curl won't create.
  • strace — Traces every syscall. strace ./conduit 9876 shows exactly what happens. Invaluable when the server "hangs."
  • lsof -i :9876 — Shows what process uses a port. Useful for EADDRINUSE errors.
  • xxd — Hex dump tool. Essential for Content-Length mismatches and encoding issues.
