TRD-00 Hints

Progressive Guidance — Conduit HTTP Server Project

Each hint has two levels. Level 1 is a nudge — enough to unblock without giving the answer. Level 2 is more direct. Try Level 1 first. If still stuck after 15+ minutes, move to Level 2.

The goal is learning, not speed. Fighting through confusion builds the mental model. These hints exist to prevent wasted time, not productive struggle.

H1: Socket Setup

Level 1 — Nudge

The three syscalls needed are socket(), setsockopt(), and bind(), in that order. Each returns -1 on failure. The headers needed are <sys/socket.h>, <netinet/in.h>, and <unistd.h>. Start by just calling socket() and printing the returned FD to confirm it works before adding anything else.

Level 2 — Stronger Push
// This is the correct signature. Fill in the struct setup yourself.
int server_fd = socket(AF_INET, SOCK_STREAM, 0);
if (server_fd == -1) {
    perror("socket");
    exit(EXIT_FAILURE);
}

AF_INET means IPv4. SOCK_STREAM means TCP (as opposed to SOCK_DGRAM for UDP). The third argument 0 lets the OS choose the protocol (which is TCP given the other two args). If this call succeeds, a valid file descriptor integer (usually 3 or higher, since 0/1/2 are stdin/stdout/stderr) is returned.

H2: SO_REUSEADDR

Level 1 — Nudge

setsockopt takes 5 arguments. The "option value" is a pointer to an int set to 1. This call must happen after socket() but before bind(). Check man 2 setsockopt for the signature.

Level 2 — Stronger Push
int opt = 1;
if (setsockopt(server_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)) == -1) {
    perror("setsockopt");
    close(server_fd);
    exit(EXIT_FAILURE);
}

&opt is a pointer to the integer value 1, meaning "enable this option." sizeof(opt) tells the kernel how many bytes that value occupies. Without this, restarting the server within ~60 seconds will fail with EADDRINUSE.

H3: The Accept Loop

Level 1 — Nudge

The structure is a while(1) loop containing accept(). The call blocks — it sits and waits until a client connects. It returns a new file descriptor for the connected client, separate from the listening FD. Think about: what happens if accept() fails? Should the whole server die, or should it skip and try again?

Level 2 — Stronger Push
while (1) {
    client_fd = accept(server_fd, ...);
    if (client_fd == -1) {
        perror("accept");
        continue;   // Don't exit! Just skip this connection.
    }
    // ... read, write, close client_fd ...
}

The second and third arguments to accept() can be NULL and NULL if the client's address information isn't needed (it isn't for TRD-00). Using continue instead of exit on a failed accept is critical — one bad connection attempt shouldn't kill the server. This is what test 11 validates.

H4: Read Behavior and Edge Cases

Level 1 — Nudge

read() returns three kinds of values: positive (number of bytes read), zero (client closed connection), and -1 (error). The code needs to handle all three. For TRD-00, only the positive case needs to proceed — the other two should close the client FD and continue.

Level 2 — Stronger Push
char buffer[4096];
ssize_t bytes_read = read(client_fd, buffer, sizeof(buffer) - 1);
if (bytes_read <= 0) {
    close(client_fd);
    continue;
}

sizeof(buffer) - 1 leaves room for a null terminator if the buffer contents are ever treated as a string. At TRD-00, the read data isn't actually used — the response is hardcoded — but reading is still necessary because some clients (including curl) won't accept a response until the server has read their request. The ssize_t type (signed size) is important: read() can return -1, so an unsigned type would misinterpret that value.

H5: Building the Response String

Level 1 — Nudge

The entire response — status line, headers, blank line, and body — can be a single string constant. Use \r\n for line endings, not just \n. The blank line between headers and body is \r\n\r\n (the last header's CRLF plus the empty line's CRLF). Think about how to get the whole thing into one write() call.

Level 2 — Stronger Push

A single const char *response = "HTTP/1.1 200 OK\r\n..."; string is the simplest approach. Use strlen() to compute the number of bytes to write. The write() call:

write(client_fd, response, strlen(response));

One detail: write() may not write all bytes in a single call (a "short write"). For TRD-00 with a small response, this is extremely unlikely, but awareness matters. Handling short writes properly becomes important in later TRDs.

H6: Byte Counting (Content-Length)

Level 1 — Nudge

The body string contains a character that looks like one glyph but is multiple bytes in UTF-8. ASCII characters are 1 byte each. Non-ASCII characters are 2–4 bytes. Don't count characters — count bytes. Use echo -n "your body string" | wc -c to verify. In C, strlen() already counts bytes, not characters.

Level 2 — Stronger Push

The em dash — (U+2014) is encoded as 3 bytes in UTF-8: 0xE2 0x80 0x94. The body Conduit is alive — TRD00 is 26 bytes. Verify:

echo -n "Conduit is alive — TRD00" | wc -c

If the source file is saved as UTF-8 (which it almost certainly is), then the C string literal will contain the 3-byte encoding, and strlen() will return 26. The Content-Length header must say 26.

Common mistake: counting characters by hand and getting 24 (treating the em dash as 1 byte). The test suite catches this by comparing the header value against the actual body byte count.

H7: Common Compilation Warnings

Level 1 — Nudge

The most common warnings with -Wall -Wextra -Werror on socket code are: unused variables, unused parameters, implicit function declarations (missing #include), and sign comparison (ssize_t vs size_t). Read each warning message carefully — GCC tells exactly what's wrong and on which line.

The typical includes for TRD-00:

#include <stdio.h>        // perror, fprintf
#include <stdlib.h>       // exit, atoi
#include <string.h>       // strlen, memset
#include <unistd.h>       // read, write, close
#include <sys/socket.h>   // socket, bind, listen, accept, setsockopt
#include <netinet/in.h>   // struct sockaddr_in, INADDR_ANY, htons

General Debugging Tips

Server starts but curl hangs: The server is probably not reading before writing. Some HTTP clients wait for the server to consume the request before accepting a response. Make sure read() is called before write().

EADDRINUSE on restart: Either a previous instance is still running (lsof -i :<port> to check, kill <pid> to fix), or SO_REUSEADDR isn't set. See H2.

Server works once, then hangs on second request: The accept loop is probably missing, or close(client_fd) isn't being called, or the code exits after the first response instead of looping.

Response body has extra garbage characters: The response string might be missing a null terminator, or write() is being told to send more bytes than the string contains. Use strlen() to compute write length, not a hardcoded number.

Tests pass but curl -v shows weird behavior: Run strace ./conduit <port> in one terminal and curl -v in another. The strace output shows every syscall, making it clear where the server is spending time or failing.
