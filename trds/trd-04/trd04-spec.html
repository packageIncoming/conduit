TRD-04: Epoll Event Loop

Conduit HTTP Server Project
Depends on: TRD-01–03  |  Difficulty: Hard  |  Estimated: 7–12 sessions

Objective

Replace the blocking single-connection accept loop with a non-blocking, edge-triggered epoll event loop. The server must handle many concurrent connections without blocking on any individual client.

⚠ This is the hardest TRD. Budget extra time. Bugs in the event loop are intermittent and difficult to reproduce. Use strace heavily — it shows exactly which syscalls the server is making and where it blocks.

Technical Requirements

R1 — Non-Blocking Sockets

Set O_NONBLOCK on the listen socket and every accepted client socket using fcntl() (or accept4() with SOCK_NONBLOCK).

Stuck? → Hints H1

R2 — Epoll Setup

Create an epoll instance with epoll_create1(0). Register the listen socket for EPOLLIN | EPOLLET.

Stuck? → Hints H1

R3 — Event Loop

Replace the accept loop with an epoll_wait() loop. Dispatch events by fd type: listen socket events call the accept handler, client socket events call the connection handler.

R4 — Edge-Triggered Accept

When the listen socket fires EPOLLIN, call accept() in a loop until EAGAIN. For each accepted fd: set non-blocking, allocate a connection_t, register with epoll for EPOLLIN | EPOLLET.

Stuck? → Hints H2

R5 — Per-Connection State

Define a connection_t struct with: fd, state enum (CONN_READING, CONN_WRITING, CONN_DONE), read buffer + length, write buffer + total length + offset. Store a pointer to it in epoll_event.data.ptr.

Stuck? → Hints H3

R6 — Read Handling

On EPOLLIN for a client: read in a loop until EAGAIN, appending to the connection's read buffer. After each read, check for \r\n\r\n. When found: parse the request (TRD-01), build the response (TRD-02/03), serialize into the connection's write buffer, transition to CONN_WRITING, and register for EPOLLOUT | EPOLLET.

Stuck? → Hints H4

R7 — Write Handling

On EPOLLOUT: write in a loop from write_buf + write_offset until all bytes are sent or EAGAIN. Track progress with write_offset. When write_offset == write_len: transition to CONN_DONE, remove from epoll, close the fd, free the connection.

Stuck? → Hints H5

R8 — Connection Cleanup

On EPOLLERR, EPOLLHUP, or any read/write error: remove from epoll, close the fd, free the connection_t and any owned buffers. No crashes, no leaked fds.

Post-Completion Comprehension

  1. What happens if you call read() only once per EPOLLIN event in edge-triggered mode?
  2. Why must the listen socket's accept handler also loop until EAGAIN?
  3. Explain the difference between storing fd vs. a connection_t * pointer in epoll_event.data. Why is the pointer approach better?
  4. What is the purpose of write_offset and what bug does it prevent?
  5. Why can't the event loop call sleep(), scanf(), or any other blocking function?

Preview: TRD-05

Currently, request processing (parsing, file reading, response building) happens on the event loop thread. If file I/O blocks (large file, slow disk), the entire event loop stalls. TRD-05 moves processing to a thread pool: the event loop dispatches completed requests to worker threads, and workers call response_send() or enqueue the response for the event loop to flush.
