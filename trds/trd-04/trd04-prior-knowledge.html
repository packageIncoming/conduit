TRD-04 Prior Knowledge

Epoll Event Loop — Conduit HTTP Server Project
Prerequisite: TRD-03 complete (response engine)  |  ⚠ Hardest TRD in Conduit

This is the most difficult milestone. The concepts are simple in isolation — non-blocking I/O, event notification, state machines — but combining them correctly produces subtle bugs that only manifest under concurrency. Read this document thoroughly before coding. Come back to it when debugging.

1   What Survives vs. What Gets Replaced

TRD-04 is the largest structural change in Conduit. Before writing any epoll code, understand what you're keeping and what you're replacing:

Keeps (unchanged) Replaced
HTTP request parser (TRD-01) — http_request_t, parsing logic The blocking accept → read → write → close loop in main()
File serving logic (TRD-02) — path resolution, MIME types, realpath() Single global read/write buffers — each connection gets its own
Response engine (TRD-03) — http_response_t, response_send() Direct read(fd) / write(fd) calls — replaced by epoll-driven state machine
Socket setup — socket(), bind(), listen(), SO_REUSEADDR Blocking accept() — replaced by non-blocking accept-in-a-loop
CLI argument parsing, docroot validation

Practical approach: Start by adding the epoll setup and event loop around the existing code. Get a single connection working through the event loop first (accept → read → process → write → close, all event-driven). Then add multi-connection support. Don't try to make everything non-blocking and concurrent in one step.

2   Blocking vs. Non-Blocking I/O

Until now, every read() and write() in Conduit has been blocking: the call waits until data is available (read) or the kernel buffer accepts the data (write). With one connection at a time, this is fine. With many connections, it's fatal — the server stalls on one slow client while others wait.

Non-blocking I/O changes the contract: if a read or write cannot complete immediately, the call returns -1 and sets errno to EAGAIN (or equivalently EWOULDBLOCK). The program is responsible for trying again later.

Setting a socket to non-blocking:

int flags = fcntl(fd, F_GETFL, 0);
fcntl(fd, F_SETFL, flags | O_NONBLOCK);

Every fd the server touches — the listen socket and every client connection — must be set to O_NONBLOCK.

3   The Epoll API

Epoll is Linux's mechanism for monitoring many file descriptors efficiently. It answers the question: "Which of my 500 open connections have data ready right now?"

Function Purpose
epoll_create1(0) Creates an epoll instance. Returns an fd that represents the epoll set.
epoll_ctl(epfd, op, fd, &ev) Modifies the interest set. op is EPOLL_CTL_ADD, _MOD, or _DEL. ev specifies which events to watch.
epoll_wait(epfd, events, max, timeout) Blocks until at least one fd is ready. Returns the number of ready events. timeout is in milliseconds (-1 = block forever).

3.1   Event Flags

FlagMeaning
EPOLLINData available to read (or new connection on listen socket)
EPOLLOUTSocket is writable (buffer space available for send)
EPOLLERRError condition on fd (always reported, no need to request)
EPOLLHUPHang-up / peer closed connection (always reported)
EPOLLETEdge-triggered mode (see §4)

3.2   The Event Loop Pattern

struct epoll_event events[MAX_EVENTS];
while (1) {
    int n = epoll_wait(epfd, events, MAX_EVENTS, -1);
    for (int i = 0; i < n; i++) {
        if (events[i].data.fd == listen_fd) {
            /* New connection(s) arriving */
            handle_accept(epfd, listen_fd);
        } else {
            /* Existing connection has activity */
            handle_connection(epfd, &events[i]);
        }
    }
}

This is the entire control flow of the server after TRD-04. There is no more accept-read-write-close loop. The event loop is the program.

4   Edge-Triggered vs. Level-Triggered

This is where most epoll bugs live.

Mode When epoll notifies Implication
Level-triggered (default) Every epoll_wait() call, as long as the condition is true (data in buffer, space to write) Simpler. If you don't read all data, you'll be notified again next loop.
Edge-triggered (EPOLLET) Once, when the condition changes (new data arrives, buffer transitions from full to not-full) More efficient but dangerous. If you don't drain all data, you get no more notifications.
The edge-triggered contract: When epoll reports EPOLLIN in ET mode, you must call read() in a loop until it returns EAGAIN. If you read once and stop, any remaining data in the kernel buffer is invisible — epoll won't notify again because no new data arrived. The connection silently hangs. This is the single most common epoll bug.

The same applies to EPOLLOUT + write(): write in a loop until EAGAIN or all data is sent.

Which mode to use: Edge-triggered is the standard choice for high-performance servers (nginx uses it). It requires more careful code but avoids redundant wakeups. Conduit should use edge-triggered.

5   The Per-Connection State Machine

Figure 1 — Per-connection state machine

Each active connection needs its own state. A typical struct:

typedef struct {
    int fd;
    int state;                /* CONN_READING, CONN_WRITING, CONN_DONE */
    char read_buf[8192];
    int read_len;             /* bytes accumulated so far */
    char *write_buf;          /* serialized response (from response engine) */
    int write_len;            /* total bytes to send */
    int write_offset;         /* bytes sent so far */
} connection_t;

The write_offset field is critical. send() on a non-blocking socket may write fewer bytes than requested. The offset tracks progress so the next EPOLLOUT event resumes from where it left off.

6   Accept Under Edge-Triggered Mode

When the listen socket gets EPOLLIN in ET mode, multiple connections may have arrived since the last notification. You must call accept() in a loop until it returns EAGAIN:

while (1) {
    int client_fd = accept(listen_fd, NULL, NULL);
    if (client_fd == -1) {
        if (errno == EAGAIN || errno == EWOULDBLOCK) break;
        perror("accept");
        break;
    }
    set_nonblocking(client_fd);
    /* Register client_fd with epoll for EPOLLIN | EPOLLET */
}

If you call accept() only once, any additional pending connections are lost until the next EPOLLIN — which in ET mode means the next new connection arrives.

7   Partial Writes

A non-blocking send()/write() may return fewer bytes than requested. This is normal — the kernel's send buffer is full. The pattern:

  1. Attempt send(fd, buf + offset, remaining, 0)
  2. If it returns n > 0: advance offset += n, reduce remaining -= n
  3. If it returns -1 with EAGAIN: stop writing, register for EPOLLOUT, wait for next notification
  4. If offset == write_len: done, transition to CONN_DONE

This is why the response engine from TRD-03 builds a complete buffer — epoll needs something to resume from.
