TRD-05 Prior Knowledge

Thread Pool — Conduit HTTP Server Project
Prerequisite: TRD-04 complete (epoll event loop with per-connection state)

1   Why a Thread Pool?

After TRD-04, the epoll event loop handles I/O multiplexing — but request processing (parsing, file I/O, response building) still runs on the event loop thread. If a file read blocks for 10ms on a slow disk, every connection stalls for 10ms. The event loop must never block.

The solution: offload processing to a fixed pool of worker threads. The event loop's only job is accepting connections, reading bytes, and flushing writes. When a complete request arrives, it's packaged as a task and handed to a worker.

2   Pthreads Fundamentals

Function Purpose
pthread_create(&tid, NULL, fn, arg) Spawn a new thread running fn(arg)
pthread_join(tid, NULL) Block until thread tid exits. Called during shutdown.
pthread_mutex_lock(&mtx) Acquire mutex. Blocks if another thread holds it.
pthread_mutex_unlock(&mtx) Release mutex.
pthread_cond_wait(&cond, &mtx) Atomically releases mutex and sleeps until signaled. Re-acquires mutex on wake.
pthread_cond_signal(&cond) Wakes one waiting thread.
pthread_cond_broadcast(&cond) Wakes all waiting threads. Used during shutdown.

Compile with -lpthread (or -pthread).

3   The Condvar Pattern

This is the most error-prone part of TRD-05. The correct pattern for a consumer (worker thread) waiting for work:

pthread_mutex_lock(&pool->mutex);
while (pool->queue_size == 0 && !pool->shutdown) {
    pthread_cond_wait(&pool->cond, &pool->mutex);
}
/* Dequeue task or exit if shutdown */
pthread_mutex_unlock(&pool->mutex);
Why while, not if? Condition variables can produce spurious wakeups — the thread wakes even though no signal was sent. POSIX explicitly allows this. If you use if, a spurious wakeup causes the worker to dequeue from an empty queue (undefined behavior or crash). The while loop re-checks the predicate after every wakeup. This is not optional.

4   Shutdown Protocol

Graceful shutdown is a coordination problem. The sequence:

  1. Set pool->shutdown = 1 (under the mutex)
  2. Call pthread_cond_broadcast() to wake all sleeping workers
  3. Call pthread_join() on each thread

Without the broadcast, sleeping workers never see the shutdown flag — they're stuck in pthread_cond_wait() forever. Without joining, the main thread may exit while workers are still running, causing undefined behavior.

Workers check the shutdown flag in their while loop. When shutdown == 1 and the queue is empty, they break out and return.

5   Task Design

A task is the unit of work passed from the event loop to a worker. At minimum it contains the connection_t * pointer — the worker needs the read buffer (to parse the request), the fd (to know where the file is served), and the write buffer (to stage the response). The event loop transfers ownership of the connection to the worker for the duration of processing.

After the worker builds the response and writes it to the connection's write buffer, it can either write directly to the fd (simpler) or hand the connection back to the event loop for EPOLLOUT-driven flushing (more correct under high load). For TRD-05, writing directly from the worker is acceptable.
