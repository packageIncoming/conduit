TRD-05: Thread Pool

Conduit HTTP Server Project
Depends on: TRD-04  |  Difficulty: Hard  |  Estimated: 5–8 sessions

Objective

Move request processing off the event loop into a fixed-size pool of worker threads using a mutex/condvar-protected task queue. The event loop enqueues; workers dequeue, process, and respond.

Technical Requirements

R1 — Thread Pool API

Implement three functions:

  • threadpool_init(int num_threads) — allocates pool, initializes mutex/condvar, spawns N worker threads
  • threadpool_enqueue(pool, task) — adds a task to the queue, signals one waiting worker
  • threadpool_destroy(pool) — sets shutdown flag, broadcasts condvar, joins all threads, frees resources
Stuck? → Hints H1

R2 — Task Queue

Implement a linked list (or ring buffer) protected by the pool's mutex. Each node holds a connection_t *. Enqueue appends to the tail; dequeue removes from the head.

Stuck? → Hints H2

R3 — Worker Thread Function

Each worker runs a loop: lock mutex → wait on condvar (with while loop checking queue empty + shutdown flag) → dequeue task → unlock → process request → send response → close connection. Workers exit when shutdown is set and the queue is empty.

Stuck? → Hints H3

R4 — Event Loop Integration

When the event loop detects a complete request (TRD-04's \r\n\r\n detection), instead of processing inline, call threadpool_enqueue() with the connection. Remove the connection's fd from epoll before enqueuing — the worker now owns it.

R5 — Worker Response Delivery

Workers call the request parser, file serving logic, and response engine from TRD-01–03. The worker writes the response directly to the connection's fd, then closes the fd and frees the connection_t.

R6 — Graceful Shutdown

On SIGTERM/SIGINT: stop accepting new connections, call threadpool_destroy() (which drains the queue and joins all workers), close the listen socket, exit cleanly with code 0.

Stuck? → Hints H4

R7 — Compilation

Link with -lpthread (or use -pthread flag). The test suite will compile with this flag.

Post-Completion Comprehension

  1. Why must pthread_cond_wait() be inside a while loop, not an if?
  2. What happens if threadpool_destroy() calls pthread_cond_signal() instead of pthread_cond_broadcast()?
  3. Why remove the connection's fd from epoll before handing it to a worker thread?
  4. If two workers dequeue simultaneously, what prevents them from getting the same task?

Preview: TRD-06

The server is now functional and concurrent but fragile. A malicious client can send a 1 GB request line, exhaust fds by opening connections without closing them, or crash the server by closing a connection mid-write. TRD-06 hardens against all of these: request size limits, connection caps, idle timeouts, SIGPIPE suppression, and disciplined error handling on every syscall.
