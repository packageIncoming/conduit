TRD-03 Hints

Progressive Guidance — Conduit HTTP Server Project
Level 1 = nudge  |  Level 2 = stronger push  |  15+ minutes before escalating

H1 — Response Struct Design (R1)

Level 1

Think about what response_send() needs to write. It needs a status code, a reason string, some number of headers, and optionally a body. The struct should hold exactly this data. For the body, you need both a pointer and a length — you can't use strlen() because binary file data (images) may contain null bytes.

Level 2
typedef struct {
    int status_code;
    char reason[32];
    struct {
        char key[128];
        char value[512];
    } headers[16];
    int header_count;
    const char *body;
    int body_length;
    int body_owned;   /* 1 = free(body) on cleanup, 0 = don't */
} http_response_t;

body_owned solves the ownership problem. Set to 1 when body is malloc'd (file data), set to 0 when body is a string literal ("Not Found").

H2 — Population API (R2)

Level 1

Each function is short — it just copies data into the struct fields. response_add_header() copies key and value into headers[header_count] and increments the count. The automatic Content-Length can be added inside response_send() right before serialization rather than during population.

Level 2
void response_set_status(http_response_t *r, int code, const char *reason) {
    r->status_code = code;
    strncpy(r->reason, reason, sizeof(r->reason) - 1);
    r->reason[sizeof(r->reason) - 1] = '\0';
}

void response_add_header(http_response_t *r, const char *key, const char *val) {
    if (r->header_count >= 16) return;  /* silent cap */
    strncpy(r->headers[r->header_count].key, key, 127);
    strncpy(r->headers[r->header_count].value, val, 511);
    r->header_count = r->header_count + 1;
}

void response_set_body(http_response_t *r, const char *data, int len, int owned) {
    r->body = data;
    r->body_length = len;
    r->body_owned = owned;
}

H3 — Serialization (R3)

Level 1

Build the response in order: status line, then each header line, then the blank line, then the body. You can either build one big buffer or write in multiple write() calls. Remember to add Content-Length from body_length — this is the automatic part the caller shouldn't have to do.

Level 2
int response_send(int fd, http_response_t *r) {
    char header_buf[4096];
    int off = 0;

    /* Status line */
    off += snprintf(header_buf + off, sizeof(header_buf) - off,
        "HTTP/1.1 %d %s\r\n", r->status_code, r->reason);

    /* Headers from struct */
    for (int i = 0; i < r->header_count; i++) {
        off += snprintf(header_buf + off, sizeof(header_buf) - off,
            "%s: %s\r\n", r->headers[i].key, r->headers[i].value);
    }

    /* Auto Content-Length */
    off += snprintf(header_buf + off, sizeof(header_buf) - off,
        "Content-Length: %d\r\n", r->body_length);

    /* Blank line */
    off += snprintf(header_buf + off, sizeof(header_buf) - off, "\r\n");

    /* Write headers */
    if (write(fd, header_buf, off) != off) return -1;

    /* Write body */
    if (r->body && r->body_length > 0) {
        if (write(fd, r->body, r->body_length) != r->body_length) return -1;
    }
    return 0;
}

Note: write() can return fewer bytes than requested even for blocking sockets under memory pressure. For TRD-03 this is rare enough to ignore. TRD-04 will handle partial writes properly.

H4 — Memory Cleanup (R4)

Level 1

The cleanup function checks the ownership flag. If the body is owned (dynamically allocated), free it. If not (string literal), leave it alone. Always reset the struct to a clean state so it can't accidentally be double-freed.

Level 2
void response_cleanup(http_response_t *r) {
    if (r->body_owned && r->body) {
        free((void *)r->body);
    }
    r->body = NULL;
    r->body_length = 0;
    r->body_owned = 0;
    r->header_count = 0;
}

The (void *) cast is needed because body is const char * and free() takes void *. Setting body = NULL after free prevents use-after-free if cleanup is accidentally called twice.
