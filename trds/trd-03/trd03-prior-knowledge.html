TRD-03 Prior Knowledge

HTTP Response Engine — Conduit HTTP Server Project
Prerequisite: TRD-02 complete (static file serving, MIME types, traversal prevention)

1   Why a Response Engine?

After TRD-02, the server has snprintf() calls scattered across every code path — one for 200, one for 404, one for 403, one for 405, one for 400. Each manually constructs the status line, headers, and body. This has three problems:

  • Duplication. Every response path repeats the same serialization logic. A bug in CRLF placement must be fixed in five places.
  • Fragility. Adding a new header (which TRD-04+ will require) means touching every code path.
  • Unclear ownership. Who allocates the body buffer? Who frees it? With ad-hoc construction, the answer varies by code path.

The response engine solves this: a struct holds the response data, a small set of functions populate it, and a single response_send() function serializes everything to the socket. Every response — success or error — flows through the same codepath.

2   Struct-Based API Design in C

C doesn't have classes, but the pattern of "struct + functions that operate on it" is the standard way to build reusable abstractions. The response engine follows this pattern:

/* The struct holds response data */
typedef struct { ... } http_response_t;

/* Functions populate it */
void response_set_status(http_response_t *resp, int code, const char *reason);
void response_add_header(http_response_t *resp, const char *key, const char *value);
void response_set_body(http_response_t *resp, const char *body, int length);

/* One function serializes and sends */
int response_send(int fd, http_response_t *resp);

Every function takes a pointer to the struct as its first argument — this is the C equivalent of this or self. The struct is initialized by the caller, populated through the API, sent, and then cleaned up.

3   Memory Ownership

This is the hardest conceptual problem in TRD-03. When the response body is a file read from disk (TRD-02), someone malloc()'d a buffer. Questions:

  • Does response_set_body() copy the data into the struct, or just store a pointer?
  • If it stores a pointer, who frees the buffer — the caller or response_send()?
  • What about error bodies like "Not Found" — those are string literals (static memory). Calling free() on them would crash.

There are two clean approaches:

Approach How it works Trade-off
Pointer + flag Store a pointer to the body and a boolean body_owned. If owned, response_send() or a cleanup function frees it. If not owned (string literal), it's left alone. Flexible, slightly more complex
Always copy response_set_body() always malloc()s and memcpy()s. A cleanup function always frees. Simpler ownership, wastes memory for small static strings

Either works. The important thing is that the rule is consistent — every code path must follow the same ownership convention, or you will have use-after-free bugs or double-free bugs. Pick one approach and enforce it everywhere.

Why this matters beyond Conduit: Memory ownership is the single most common source of bugs in C/C++ systems code. In a FAANG systems interview, being able to articulate "this struct owns its body buffer and frees it in the cleanup function" demonstrates the kind of thinking that separates junior from mid-level systems engineers.

4   Serialization to Wire Format

The response on the wire looks like:

HTTP/1.1 200 OK\r\n
Content-Type: text/html\r\n
Content-Length: 45\r\n
\r\n
<html><body>Hello</body></html>

response_send() must produce this byte sequence from the struct. The order is fixed: status line → headers → blank line → body. The blank line (\r\n) separating headers from body is mandatory and easy to forget.

Two implementation strategies:

  • Build one big buffer with snprintf(), then write() once. Simple, but requires knowing the total size upfront.
  • Write in pieces: write() the status line, then each header, then \r\n, then the body. More write() calls but no allocation needed for the serialized headers.

For TRD-03 (still single-threaded, blocking I/O), either works. TRD-04 will impose constraints that favor buffering the full response — but don't over-engineer for that now.
