TRD-03: HTTP Response Engine

Conduit HTTP Server Project
Depends on: TRD-01, TRD-02  |  Difficulty: Medium  |  Estimated: 3–5 sessions

Objective

Extract all response construction into a reusable http_response_t API. Every HTTP response — success or error — must flow through a single serialization codepath. Refactor all existing TRD-01/02 response logic to use the engine.

Technical Requirements

R1 — Response Struct

Define http_response_t containing: status code (int), reason phrase (string), headers (key-value array with count), body pointer, body length, and an ownership flag or strategy for the body buffer.

Stuck? → Hints H1

R2 — Population API

Implement at minimum:

  • response_set_status(resp, code, reason) — sets status code and reason phrase
  • response_add_header(resp, key, value) — appends a header to the headers array
  • response_set_body(resp, data, length) — attaches a body with its byte length

Content-Length must be set automatically from the body length when a body is present. Do not require the caller to add it manually.

Stuck? → Hints H2

R3 — Serialization and Send

Implement response_send(fd, resp). It serializes the struct to HTTP wire format (status line → headers → \r\n → body) and writes to the socket fd. Returns 0 on success, -1 on write failure.

Stuck? → Hints H3

R4 — Memory Cleanup

Implement response_cleanup(resp) or equivalent. Frees any dynamically allocated body buffer according to the ownership strategy. Must be safe to call on responses with static string bodies (string literals must not be freed).

Stuck? → Hints H4

R5 — Refactor All Response Paths

Remove all existing ad-hoc snprintf() response construction from TRD-01/02 code. Every response (200, 400, 403, 404, 405) must use the response engine. Add 500 Internal Server Error for unexpected failures (e.g., read() error mid-file, malloc failure).

R6 — Error Response Bodies

All error responses (400, 403, 404, 405, 500) must include a short plain-text body describing the error (e.g., "Not Found", "Method Not Allowed"). The body must not be empty.

Post-Completion Comprehension

  1. What happens if response_cleanup() calls free() on a string literal? How does the ownership flag prevent this?
  2. Why should Content-Length be set automatically inside the engine rather than by the caller?
  3. If write() returns fewer bytes than the full response, what happened? (Think about what TRD-04 will need to handle.)
  4. When would the server legitimately return 500 instead of 400 or 404?

Preview: TRD-04

The server currently handles one connection at a time — blocking on read() and write(). TRD-04 replaces this with a non-blocking epoll event loop where the response engine's output gets buffered per-connection and flushed incrementally as the socket becomes writable. The response struct and serialization from TRD-03 become the foundation that epoll writes from.
