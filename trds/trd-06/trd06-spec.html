TRD-06: Hardening

Conduit HTTP Server Project
Depends on: TRD-01–05  |  Difficulty: Medium-Hard  |  Estimated: 4–6 sessions

Objective

Make the server robust against malformed input, resource exhaustion, and ungraceful client behavior. After this TRD, the server should not crash under any hostile condition.

Technical Requirements

R1 — SIGPIPE Suppression

Register signal(SIGPIPE, SIG_IGN) at startup. Verify that write() to a closed socket returns -1/EPIPE instead of killing the process.

Stuck? → Hints H1

R2 — Request Size Limit

If the read buffer fills (8 KB) without finding \r\n\r\n, return 400 Bad Request and close the connection. Do not silently drop — the client must receive an error response.

Stuck? → Hints H2

R3 — Connection Count Tracking

Maintain an atomic or mutex-protected counter of active connections. When at capacity, stop calling accept() until connections close. Log when the limit is hit.

R4 — Idle Connection Timeout

Store a last_active timestamp on each connection. Use epoll_wait()'s timeout parameter (e.g., 1000ms) to periodically sweep and close connections idle longer than a threshold (default: 30 seconds, configurable via #define).

Stuck? → Hints H3

R5 — Signal Handler

Install a SIGINT/SIGTERM handler using sigaction() that sets a volatile sig_atomic_t flag. The event loop checks this flag each iteration and initiates graceful shutdown. No non-async-signal-safe calls in the handler.

R6 — Error Discipline

Every read(), write(), send(), open(), accept(), malloc() return value must be checked. No silent failures. Connection-level errors close that connection — they must never crash the server.

Post-Completion Comprehension

  1. Why is SIGPIPE fatal by default? What design assumption does that reflect?
  2. Why must the signal handler use volatile sig_atomic_t instead of a plain int?
  3. How does the idle timeout sweep interact with the epoll event loop without blocking?
  4. What is a slowloris attack and which requirement in this TRD defends against it?
