TRD-06 Prior Knowledge

Hardening — Conduit HTTP Server Project
Prerequisite: TRD-05 complete (thread pool + epoll event loop)

The server is functional and concurrent. Now it needs to not die under hostile conditions. Every concept in this document addresses a real class of production server bugs.

1   SIGPIPE

When the server calls write() on a socket whose remote end has already closed, the kernel delivers SIGPIPE. The default action for SIGPIPE is to terminate the process. One disconnected client kills the entire server.

Two suppression methods:

MethodHow
signal(SIGPIPE, SIG_IGN) At startup, globally ignore SIGPIPE. write() then returns -1 with errno == EPIPE instead of killing the process. Simplest approach.
MSG_NOSIGNAL flag Pass MSG_NOSIGNAL as a flag to send() (not write()). Suppresses SIGPIPE per-call. More granular but requires changing every write to use send().

For Conduit, signal(SIGPIPE, SIG_IGN) at startup is the right choice — global, one line, impossible to forget.

2   Request Size Limits

Without a size limit, a malicious client can send a multi-gigabyte request line that exhausts the server's memory. The read buffer in connection_t (from TRD-04) already has a fixed size — if it fills up before \r\n\r\n is found, the request is oversized. Return 400 Bad Request and close the connection.

8 KB is a common limit for request headers (Apache and nginx both default to this range). This is already implicit in the 8192-byte read buffer from TRD-04 — TRD-06 makes it explicit and ensures the server responds with 400 rather than silently dropping the connection.

3   Connection Limits

Every open connection consumes a file descriptor. Linux has per-process fd limits (typically 1024 soft, 65536 hard). If the server runs out, accept() fails with EMFILE and new clients are refused with no response at all.

A responsible server tracks open connections and stops accepting when at capacity, optionally returning 503 Service Unavailable. This is better than silently dropping connections at the kernel level.

4   Idle Connection Timeouts

A client may connect, send partial headers, and then go silent — intentionally (slowloris attack) or accidentally (network failure). Without timeouts, these connections sit in CONN_READING forever, consuming memory and fds.

Implementation pattern:

  1. Store a last_active timestamp (from time() or clock_gettime()) in each connection_t
  2. Update it on every successful read() or write()
  3. Pass a timeout to epoll_wait() (e.g., 1000ms) so it returns periodically even with no events
  4. On each epoll_wait() return, sweep all connections: if now - last_active > threshold, close the connection
Why not sleep() or alarm()? sleep() blocks the event loop — fatal. alarm() uses signals which are messy with threads. The epoll_wait() timeout is the correct mechanism: it returns periodically without blocking, and the sweep runs in the event loop's normal flow.

5   Signal Handling with volatile sig_atomic_t

TRD-05 introduced SIGTERM shutdown. TRD-06 formalizes the signal handler. The handler must only set a flag — no printf(), no malloc(), no mutex operations. Only async-signal-safe functions may be called inside a signal handler.

volatile sig_atomic_t shutdown_flag = 0;

void handle_signal(int sig) {
    (void)sig;
    shutdown_flag = 1;
}

volatile prevents the compiler from optimizing away reads of the flag. sig_atomic_t guarantees atomic reads/writes. The event loop checks shutdown_flag on each iteration.
