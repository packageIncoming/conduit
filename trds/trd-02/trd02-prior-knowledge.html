TRD-02 Prior Knowledge

Static File Serving — Conduit HTTP Server Project
Prerequisite: TRD-01 complete (request parsing, method/path routing, status codes)

1   POSIX File I/O

Serving a file means: open it, figure out how big it is, read its contents into a buffer, and write that buffer as the HTTP response body. Four syscalls handle this.

Syscall Purpose Key detail
open(path, O_RDONLY) Opens the file, returns a file descriptor Returns -1 on failure (file not found, permissions). Check errno.
fstat(fd, &sb) Gets file metadata into a struct stat sb.st_size gives the file size in bytes — use this for Content-Length.
read(fd, buf, count) Reads bytes from the file into a buffer May return fewer bytes than requested. Loop until all st_size bytes are read or EOF.
close(fd) Releases the file descriptor Always close — even on error paths. Leaked file descriptors exhaust the OS limit.

The pattern is always: open → fstat → allocate buffer → read → use data → close. If open() fails with ENOENT, the file doesn't exist — return 404. If it fails with EACCES, return 403.

2   Path Resolution

The server receives a URI path like /style.css from the parsed request (TRD-01). This must be mapped to a real filesystem path. The mapping is:

filesystem_path = docroot + request_path

Where docroot is a directory specified on the command line. If docroot is /srv/www and the request is GET /style.css, the server opens /srv/www/style.css.

Special case: GET / should serve index.html from the document root. This is a convention, not a spec requirement — but it's universal enough that omitting it would be surprising.

2.1   Directory Traversal

This is the security-critical concept in TRD-02.

Figure 1 — Directory traversal attack and defense

If the server naively concatenates docroot + path, an attacker can use .. sequences to escape the document root and read any file on the system. This is a real CVE-class vulnerability — it appears regularly in web server security advisories.

The defense has two steps:

  1. Resolve the full path using realpath(), which resolves all .., symlinks, and relative components into an absolute canonical path.
  2. Verify the resolved path starts with the document root. If realpath() returns /etc/passwd and the docroot is /srv/www, the prefix check fails → return 403.
Why not just check for ".." in the raw URI? Because attackers can encode it: %2e%2e, double-encoding, Unicode tricks, and symlinks all bypass string-level checks. realpath() resolves the actual filesystem path at the OS level — it handles all of these cases. Always resolve first, then check the prefix.

3   MIME Types

The Content-Type header tells the client how to interpret the response body. It's determined by the file extension.

ExtensionContent-TypeNotes
.htmltext/htmlBrowsers render this as a web page
.csstext/cssBrowsers reject CSS served with wrong type
.jsapplication/javascripttext/javascript also accepted
.txttext/plainDisplayed as raw text
.jpgimage/jpegNote: extension is .jpg but type is jpeg
.pngimage/png
(unknown)application/octet-streamSafe fallback — browser will offer download

Implementation: find the last . in the filename with strrchr(path, '.'), then compare the extension against a lookup table. If no dot or no match, use the octet-stream fallback.

4   The New CLI Interface

TRD-01's server took one argument: ./conduit <port>. TRD-02 adds a second: ./conduit <port> <docroot>. If either argument is missing, print a usage message to stderr and exit with code 1. Validate that the docroot directory exists (opendir() or stat()) before entering the accept loop.
