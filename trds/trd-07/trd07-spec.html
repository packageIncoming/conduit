TRD-07: Benchmarking & Documentation

Conduit HTTP Server Project — Final TRD
Depends on: TRD-01–06  |  Difficulty: Low-Medium  |  Estimated: 3–5 sessions

Objective

Measure the server's performance honestly, verify it doesn't leak memory, write a professional README, and finalize the repository as a portfolio artifact.

Deliverables

D1 — Benchmark Data

Run wrk against Conduit at concurrency levels 1, 10, 50, and 100. Record req/sec and p99 latency for each. Run 3 trials per level, report the median. Include a warm-up run before recording.

Stuck? → Hints H1

D2 — Baseline Comparison

Run the same wrk tests against Python http.server serving the same file. Produce a comparison table. Frame results honestly (see Prior Knowledge §5).

D3 — Valgrind Memcheck

Run the server under valgrind --leak-check=full. Send a mix of valid and invalid requests. Send SIGTERM. Confirm 0 bytes definitely lost. Fix any leaks found.

Stuck? → Hints H2

D4 — README.md

Write a README with these sections:

  1. Project Summary — what Conduit is, what it does, tech stack
  2. Architecture — event loop, thread pool, request pipeline. Include a diagram.
  3. Build & Run — exact commands to compile and start the server
  4. Benchmark Results — table with req/sec and p99 at each concurrency level, comparison to baseline, hardware specs, methodology notes
  5. Design Decisions — why epoll, why edge-triggered, why fixed thread pool, why the response engine abstraction
  6. Known Limitations — what Conduit does not support (pipelining, TLS, chunked encoding, HTTP/2, etc.)

D5 — Repository Cleanup

Confirm the repo is clean: no build artifacts committed, .gitignore present, code compiles from a fresh clone, README renders correctly on GitHub. Tag the final commit.

Post-Completion Comprehension

  1. Why does p99 latency matter more than average latency for evaluating a server?
  2. Why would Conduit's req/sec plateau or decrease at very high concurrency levels?
  3. What does valgrind's "still reachable" category mean, and why is it usually acceptable at exit?
  4. Why do benchmarks over loopback produce unrealistically high req/sec numbers?

Project Complete

There is no TRD-08. Conduit is a finished HTTP/1.1 server with epoll I/O multiplexing, a thread pool, security hardening, honest benchmarks, and professional documentation. It demonstrates: POSIX socket programming, non-blocking I/O, Linux-specific APIs (epoll), concurrent systems design (pthreads), memory management in C, and the ability to measure and communicate technical results. This is the project to talk about in systems engineering interviews.
