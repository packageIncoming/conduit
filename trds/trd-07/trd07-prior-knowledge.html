TRD-07 Prior Knowledge

Benchmarking & Documentation — Conduit HTTP Server Project
Prerequisite: TRD-06 complete (hardened server)  |  Final TRD

No new server code. This TRD is about measuring what was built, checking it doesn't leak, and writing the documentation that makes it a portfolio artifact. The deliverables are data, prose, and a clean repo — not a new feature.

1   Benchmarking with wrk

wrk is an HTTP benchmarking tool. It opens many connections to the server, sends requests as fast as possible, and reports throughput and latency.

wrk -t2 -c10 -d10s http://localhost:9876/index.html
FlagMeaning
-t2Use 2 threads inside wrk (not the server's threads)
-c10Keep 10 connections open simultaneously
-d10sRun for 10 seconds

Key metrics in the output:

  • Requests/sec — throughput. Higher is better.
  • Latency (Avg, p50, p99) — response time. Lower is better. p99 means "99% of requests completed within this time." p99 matters more than average for servers because outliers indicate performance cliffs.
  • Transfer/sec — bytes throughput. Depends on response size, less useful for comparison.

1.1   Methodology

  • Warm up — run wrk once for 5 seconds before recording. The first run fills caches and stabilizes thread pools.
  • Multiple trials — run each configuration 3 times, report the median.
  • Control variables — same machine, same file, same wrk thread count. Only vary connection count.
  • Concurrency levels — test at 1, 10, 50, and 100 connections to show how the server scales.
Install wrk: On Ubuntu/Debian: sudo apt install wrk. On macOS: brew install wrk. From source: github.com/wg/wrk.

2   Comparison Baseline

Benchmarking in isolation means nothing — numbers need context. Compare Conduit against Python's built-in http.server serving the same file:

cd /path/to/docroot && python3 -m http.server 8080

Then run the same wrk tests against http://localhost:8080/index.html. Python's server is single-threaded and blocking — Conduit should outperform it significantly. This comparison is fair and demonstrates real gains.

A comparison against nginx is optional and educational, but must be framed honestly: nginx is a production-grade server with decades of optimization. Conduit losing to nginx at high concurrency is expected and fine.

3   Valgrind Memory Audit

valgrind --leak-check=full --show-leak-kinds=all ./conduit 9876 ./docroot

Send several requests (mix of 200, 404, 405), then send SIGTERM. Check valgrind's summary:

  • "definitely lost: 0 bytes" — the target. No memory that was allocated and never freed.
  • "still reachable" — memory still pointed to at exit. Usually harmless (global data, not-yet-freed pools). Acceptable for Conduit.
  • "indirectly lost" — memory lost because the pointer to it was lost. Fix these — they indicate a real leak chain.
Install valgrind: sudo apt install valgrind. Note: valgrind slows the server ~10–20x. Use short benchmarks and low concurrency when testing under valgrind.

4   README Structure

A systems project README is a technical document, not marketing copy. Required sections:

SectionContent
Project SummaryOne paragraph: what it is, what it does, what technologies it uses.
ArchitectureHigh-level description of the event loop, thread pool, and request pipeline. An ASCII or SVG diagram is ideal.
Build & Rungcc command, how to run, command-line arguments.
Benchmark ResultsTable of req/sec and p99 at each concurrency level. Methodology notes. Comparison baseline. Hardware specs.
Design DecisionsWhy epoll over poll/select. Why edge-triggered. Why fixed thread pool. These show engineering thinking.
Known LimitationsWhat it doesn't do (HTTP/1.1 pipelining, TLS, chunked encoding, etc.). Honesty is more impressive than omission.

5   Honest Framing

Benchmark results on a dev laptop over loopback eliminate network latency entirely, inflating req/sec. Frame results as:

"Throughput measured under N concurrent connections on local loopback, [CPU model], [RAM]. These numbers reflect server-side processing capacity, not real-world performance over a network."

Recruiters who know systems will respect this framing. Inflated claims ("10x faster than nginx") will be spotted and damage credibility.
